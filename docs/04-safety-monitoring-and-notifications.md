# Safety Monitoring And Notifications

## Safety Scope

The project monitors three categories of household risk:

- medication adherence context
- unfamiliar human presence
- physical hazards

These are not random features. They map directly to common family concerns in Alzheimer's care:

- Was the medicine identified and scheduled?
- Is someone near the patient who should not be ignored?
- Is there an immediate danger in the room?

## Familiar And Unfamiliar Face Logic

The familiar-face system is implemented in [backend/processing/face_manager.py](/backend/processing/face_manager.py).

The workflow is:

1. detect faces in the frame
2. encode them with `face_recognition`
3. compare the embedding to known stored encodings
4. update existing records when a match is found
5. create a new unknown-face record when no match exists
6. promote a face to familiar after enough repeated sightings

This is a pragmatic care-oriented model:

- repeated exposure builds familiarity over time
- the system does not require a perfect pre-labeled face set to start being useful

## Unknown Visitor Alerts

When a new face appears in an environment that already has familiar faces, the system flags that face as unfamiliar and creates an alert.

The alert includes:

- a warning severity
- explanatory copy
- the associated face image when available
- delivery routing to both patient and caregiver channels

### Notification Delivery For Unknown Visitors

When an unfamiliar face is detected, the system dispatches notifications through all configured channels:

- **SMS to the patient**: "Unknown person detected nearby"
- **SMS to the caregiver**: "Unknown person detected near {patient_name}"
- **Email to the patient**: warning email with the face image embedded when available
- **Email to the caregiver**: identical warning email with escalation context

Each channel has an independent cooldown key (`face:{patient_id}:{face_id}:patient`, `face:{patient_id}:{face_id}:caregiver`, `face:{patient_id}:{face_id}:patient-email`, `face:{patient_id}:{face_id}:caregiver-email`), so suppressing a duplicate SMS does not suppress the corresponding email.

## Hazard Detection

Hazard detection currently combines:

- `YOLOv8` for knives and scissors
- `Roboflow` hosted models for fire and gun detection

The backend normalizes hazard classes into care-oriented labels such as:

- `sharp object`
- `weapon`
- `fire`

This is a product-quality decision. It keeps downstream messaging clean and understandable.

## Hazard Escalation

Hazard detections are treated as critical events.

The system:

- captures an image crop when possible
- stores an alert record in Supabase with severity `critical` and type `hazard_sos`
- dispatches notifications through all channels

### Hazard Notification Dispatch

| Channel | Patient | Caregiver |
|---|---|---|
| SMS | short warning (e.g., "Warning: fire detected") | escalation copy (e.g., "SOS: Fire detected near {patient_name}. 911 should be called.") |
| MMS | not used | escalation copy with hazard crop image when available |
| Email | alert with subject, body, and image link when available | identical alert with escalation copy and image evidence |

The hazard message templates are specific to the hazard type:

- **Fire**: "Warning: fire detected" / "SOS: Fire detected near {patient_name}. 911 should be called."
- **Weapon**: "Warning: weapon detected" / "SOS: Weapon detected near {patient_name}. 911 should be called."
- **Sharp object**: "Warning: sharp object detected" / "SOS: Sharp object near {patient_name}. 911 should be called."

### Email Format For Hazard Alerts

The email for hazard alerts is generated by `SendGridEmailService._default_html` and includes:

- a styled HTML layout with the subject as the heading
- the alert body rendered as paragraphs
- a clickable "View captured image" link to the hazard crop stored in Supabase
- an inline `<img>` tag rendering the same crop directly in the email body
- all dynamic content is HTML-escaped to prevent injection
- a plain-text fallback is always included for email clients that do not render HTML

## Notification Channels

### SMS And MMS (Twilio)

The implemented phone notification path is based on Twilio:

- SMS for short text alerts
- MMS for image-backed escalations

The backend has soft-fail handling, so a Twilio outage does not take down the frame-processing loop. If a send fails, the error is logged and processing continues.

### Email (SendGrid)

Email is a first-class notification channel, not a secondary fallback.

The email service is implemented in [backend/services/email_service.py](/backend/services/email_service.py) using the SendGrid Mail Send v3 API over `urllib.request` with no additional SDK dependency.

Key implementation details:

- **Dual-format delivery**: every email includes both `text/plain` and `text/html` content types
- **Automatic HTML generation**: when no explicit `html_body` is provided, the service generates a styled HTML template with the subject as a heading, the body as paragraphs, and an optional image block
- **Image evidence**: when an `image_url` is available (from Supabase storage), the HTML includes both a clickable link and an inline `<img>` tag with responsive styling
- **Cooldown isolation**: the email service maintains its own cooldown dictionary, separate from the Twilio cooldowns, with configurable windows per event
- **Graceful degradation**: when `SENDGRID_API_KEY` or `SENDGRID_FROM_EMAIL` is not configured, the service logs a dry-run message and returns `"dry-run-email"` instead of failing
- **TLS handling**: uses `certifi` for CA certificate resolution when available, falling back to system defaults
- **Reply-to support**: an optional `SENDGRID_REPLY_TO` address can be configured so recipients can respond to alerts

### Configuring SendGrid

To enable email delivery:

1. create a SendGrid account and generate an API key with Mail Send permission
2. verify a Single Sender Identity (e.g., `vishruth.nm@gmail.com`) or authenticate a domain
3. set the following environment variables in `.env`:

```
SENDGRID_API_KEY=<your-sendgrid-api-key>
SENDGRID_FROM_EMAIL=vishruth.nm@gmail.com
SENDGRID_FROM_NAME=VALIANT
SENDGRID_REPLY_TO=
```

When both `SENDGRID_API_KEY` and `SENDGRID_FROM_EMAIL` are present, email delivery is live. When either is missing, the service operates in dry-run mode and logs what would have been sent.

## Medicine Reminder Notifications

When APScheduler fires a medicine reminder, the scheduler in [backend/services/scheduler.py](/backend/services/scheduler.py) dispatches through all channels:

- **SMS to the patient**: "Time to take {medicine_name} {dosage}"
- **SMS to the caregiver**: "Reminder: {patient_name} should take {medicine_name} {dosage}"
- **Email to the patient**: subject "Medicine reminder: {medicine_name}", body identical to the SMS text
- **Email to the caregiver**: subject "Reminder for {patient_name}: {medicine_name}", body identical to the SMS text

Each delivery uses a unique cooldown key:

- `reminder:patient:{reminder_id}` for patient SMS
- `reminder:caregiver:{reminder_id}` for caregiver SMS
- `reminder-email:patient:{reminder_id}` for patient email
- `reminder-email:caregiver:{reminder_id}` for caregiver email

This means each channel is independently rate-limited. If a reminder SMS was already sent within the cooldown window, the email can still go through, and vice versa.

## Email And Phone Capture During Onboarding

Email addresses are captured during patient onboarding through the `AuthGate` component in [frontend/src/components/AuthGate.tsx](/frontend/src/components/AuthGate.tsx).

The onboarding form collects:

- patient name (required)
- patient phone number (required)
- patient email (optional)
- caregiver name (optional)
- caregiver phone number (required)
- caregiver email (optional)

Email validation happens at two layers:

1. **Frontend**: [frontend/src/lib/email.ts](/frontend/src/lib/email.ts) normalizes and validates the email format before form submission
2. **Backend**: [backend/utils/email.py](/backend/utils/email.py) applies a strict regex pattern and rejects malformed addresses; the `POST /api/patient` endpoint returns a `422` if an email was provided but is invalid

If no email is provided, the notification system simply skips the email channel for that recipient. This means the system works with phone-only onboarding and gains email capability when addresses are provided.

## Dashboard Monitoring

The frontend provides a live operational view of:

- current patient session
- medicine schedule
- known and unknown faces
- recent alerts with acknowledgment controls
- camera preview with server-side bounding boxes

This is crucial because notifications alone are not enough. Caregivers also need a place to inspect the system's current state.

## Multi-Camera Network Monitoring

The companion multi-camera server on the `network` branch adds a second monitoring surface:

- any phone on the local network can act as a surveillance camera
- the phone registers with a human-readable name and streams frames via HTTP
- motion detection runs on the phone and triggers real-time dashboard alerts
- the network dashboard shows all cameras in a grid with live status, frame updates, and motion event history

This extends the safety monitoring perimeter from a single laptop webcam to any number of phones positioned around the home.

## Why The Overlay Matters

The camera preview now includes bounding boxes for:

- medicines
- faces
- hazards

That serves two purposes:

- it helps users understand what the system thinks it is seeing
- it gives immediate visual trust in the inference pipeline

For safety software, interpretability is not optional. It is part of usability.

## Data As Evidence

The system does not only raise alerts. It also stores evidence:

- medicine crop images
- hazard images
- face images
- timestamps
- acknowledgment state
- delivery status for both patient and caregiver channels

That gives the project the beginnings of a real audit trail.

## Notification Delivery Resilience

The notification layer is designed to be resilient across all channels:

- **Twilio SMS/MMS failures**: caught and logged; the processing loop continues
- **SendGrid email failures**: caught and logged with HTTP status and response body; SMS delivery is unaffected
- **Missing credentials**: each service independently detects missing configuration and enters dry-run mode without affecting the other
- **Cooldown isolation**: email and SMS cooldowns are tracked in separate dictionaries so a cooldown on one channel never suppresses the other
- **Roboflow authorization failures**: trigger a one-hour backoff to prevent wasteful retries

## Safety Monitoring Summary

The system's safety model is effective because it does not treat every event equally.

- medicines become reminders delivered via SMS and email
- unfamiliar people become warnings delivered via SMS and email
- hazards become escalations delivered via SMS, MMS, and email with image evidence

That event taxonomy, combined with multi-channel notification delivery, is one of the strongest product decisions in the current build.
